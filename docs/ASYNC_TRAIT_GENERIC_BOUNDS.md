# `async_trait` Generic Bounds and Pipeline Error Types

This note explains why generic `async_trait` implementations often need bounds like:

```rust
E: From<crate::error::Error> + Send + Sync + 'static
```

and how that fits the service-layer pipeline pattern used in this repository.

## The short version

When `#[async_trait]` is used on a trait method, the async method is transformed into a boxed future. That future captures:

- `self`
- method arguments
- local variables
- output types such as `StepAction<E>`

Because of that transformation, generic types used by the method often need extra bounds so the boxed future can be valid.

## Conceptual desugaring

This trait:

```rust
#[async_trait::async_trait]
pub trait PipelineStep<T, E> {
    fn name(&self) -> &'static str;
    fn should_execute(&self, context: &T) -> bool;
    async fn execute(&self, context: &mut T) -> StepAction<E>;
}
```

is conceptually turned into something like:

```rust
pub trait PipelineStep<T, E> {
    fn name(&self) -> &'static str;
    fn should_execute(&self, context: &T) -> bool;

    fn execute<'a>(
        &'a self,
        context: &'a mut T,
    ) -> core::pin::Pin<
        Box<dyn core::future::Future<Output = StepAction<E>> + Send + 'a>
    >;
}
```

So the future returned by `execute()` contains `StepAction<E>` and also captures references to `self` and `context`.

## Why the bounds appear

### `E: From<crate::error::Error>`

This is needed when a step creates a concrete service error and converts it with `.into()`:

```rust
StepAction::Abort(crate::error::Error::SettingsError("...".to_string()).into())
```

That means the generic pipeline error type `E` must be constructible from `crate::error::Error`.

### `E: Send`

The boxed future generated by `async_trait` is commonly `Send`. If the future output is:

```rust
StepAction<E>
```

then `E` also needs to be compatible with that `Send` future.

### `E: Sync`

This is commonly required alongside `Send` in async trait bounds. It keeps the generic type safe to share in concurrent async contexts and avoids friction with the generated future type.

### `E: 'static`

This means `E` must not contain short-lived borrowed references. In practice:

- `String` is fine
- `Vec<String>` is fine
- borrowed local data like `&str` tied to a temporary scope is not

This is common with boxed async futures because the erased future type is easier to use when captured/output types do not borrow temporary stack data.

### `T: Send + Sync`

If the async method captures `&mut T` or `&T`, and the generated future is `Send`, then the context type also often needs `Send + Sync`.

## Recommended pipeline pattern

Use a generic `StepAction<E>`:

```rust
#[derive(Debug, Clone, PartialEq)]
pub enum StepAction<E> {
    Continue,
    Skip,
    Abort(E),
}
```

But keep **one unified error type per pipeline**.

For example:

```rust
pub enum LibretroPreflightError {
    UnsupportedExtension(String),
    MissingFirmware(Vec<String>),
    NoFileInFileSet,
    SystemDirNotSet,
    DownloadError(crate::error::Error),
}
```

Each step maps its local failure into that single pipeline error type. Avoid making every step invent a completely unrelated error payload.

## Recommended `async_trait` impl shape

```rust
#[async_trait::async_trait]
impl<T, E> PipelineStep<T, E> for ConnectToCloudStep<T, E>
where
    T: CloudConnectionContext + Send + Sync,
    E: From<crate::error::Error> + Send + Sync + 'static,
{
    fn name(&self) -> &'static str {
        "connect_to_cloud"
    }

    fn should_execute(&self, context: &T) -> bool {
        context.should_connect()
    }

    async fn execute(&self, context: &mut T) -> StepAction<E> {
        let settings = match context.settings().s3_settings.clone() {
            Some(settings) => settings,
            None => {
                return StepAction::Abort(
                    crate::error::Error::SettingsError("S3 settings missing".to_string()).into(),
                );
            }
        };

        let credentials = match context.settings_service().load_credentials().await {
            Ok(Some(creds)) => creds,
            Ok(None) => {
                return StepAction::Abort(
                    crate::error::Error::SettingsError("S3 credentials missing".to_string())
                        .into(),
                );
            }
            Err(e) => {
                return StepAction::Abort(
                    crate::error::Error::SettingsError(format!(
                        "Failed to get S3 credentials: {}",
                        e
                    ))
                    .into(),
                );
            }
        };

        match cloud_storage::S3CloudStorage::get_bucket(
            settings.endpoint.as_str(),
            settings.region.as_str(),
            settings.bucket.as_str(),
            credentials.access_key_id.as_str(),
            credentials.secret_access_key.as_str(),
        ) {
            Ok(cloud_ops) => {
                *context.cloud_ops_mut() = Some(std::sync::Arc::new(cloud_ops));
                StepAction::Continue
            }
            Err(e) => StepAction::Abort(
                crate::error::Error::CloudSyncError(format!(
                    "Failed to connect to S3: {}",
                    e
                ))
                .into(),
            ),
        }
    }
}
```

## `should_execute()` vs `execute()`

For pipeline steps in this repository:

- conditions that determine whether a step is valid to run belong in `should_execute()`
- `execute()` may use `expect(...)` for invariants already guaranteed by `should_execute()`
- real business failures that can still occur during a valid step should return `StepAction::Abort(...)`

Example:

```rust
fn should_execute(&self, context: &PrepareLaunchContext) -> bool {
    context.state.selected_file.is_none() && context.state.download_results.is_some()
}

async fn execute(
    &self,
    context: &mut PrepareLaunchContext,
) -> StepAction<LibretroPreflightError> {
    let download_results = context
        .state
        .download_results
        .as_ref()
        .expect("download_results are checked in should_execute");

    // real business failure:
    // no files in the file set
    match download_results.output_file_names.first() {
        Some(name) => {
            context.state.selected_file = Some(name.clone());
            StepAction::Continue
        }
        None => StepAction::Abort(LibretroPreflightError::NoFileInFileSet),
    }
}
```

## Practical guidance

1. Use `StepAction<E>`, not `StepAction<Error>`, in generic pipeline traits.
2. Convert concrete step errors with `.into()`.
3. Prefer one pipeline-specific error enum per pipeline.
4. Keep `should_execute()` responsible for step invariants.
5. Use `expect(...)` in `execute()` only when `should_execute()` already proved the value exists.
6. Return `StepAction::Abort(...)` only for real domain or workflow failures.
7. Prefer owned error data (`String`, `Vec<String>`) over borrowed references in async pipeline errors.

## Rule of thumb

If a generic `async_trait` impl fails to compile, check whether the generic type:

1. appears in the async output
2. is converted from a concrete error using `.into()`
3. is captured by the async body

If so, bounds like these are often the right fix:

```rust
T: Send + Sync
E: From<crate::error::Error> + Send + Sync + 'static
```
